# Boomi × Firebase 農業DX API統合デモ ― 実演シナリオ

## デモの目的

Firebaseで既にAPIを公開しているお客様（製造・農業分野）に対し、
**Boomiを「Firebaseの前」に置くことでガバナンスと開発効率を同時に手に入れる**構成を、
実際の画面操作で体感していただく。

> **伝え方の原則**: Firebaseを否定するのではなく、「その価値を最大化するためのツール」としてBoomiを位置づける。

## デモの流れ（約20〜25分）

### パート1: 課題の共有（約3分）

**スライド1〜2を使用**

お客様の「あるある」を3つ提示し、共感を得る。

- **既存資産の塩漬け**: 以前作った手作りAPIが再利用できず、埋もれている
- **運用の不透明さ**: FirebaseでAPIは作れるが、「誰が・いつ・どれだけ使っているか」がわからない
- **拡張の壁**: 気象データやセンサーなど新しい外部データを繋ぐたびに、Firebase側でNode.jsのコードを書き足す必要がある

> **話し方のポイント**: 「Firebase自体は素晴らしい開発環境です。ただ、APIが増えてくると…」と、肯定から入る。

---

### パート2: ソリューション概要（約3分）

**スライド3（アーキテクチャ図）を使用**

Boomiの役割を「門番」と「仲介役」に例えて説明する。

| Boomiの機能 | 役割 | 一言で言うと |
|------------|------|-----------|
| API Gateway（APIM） | Firebaseの前に立つ「門番」 | 認証・流量制限を後付け |
| Integration Engine（iPaaS） | システム間の「仲介役」 | コード不要でデータ変換・連携 |
| 統一エンドポイント | 利用者への「窓口」 | バラバラなURLを一つに集約 |

**キーメッセージ**:
「Firebaseは『ビジネスロジック』に集中させ、他システムとのデータ同期やフォーマット変換はBoomiのローコード画面で行います。」

---

### パート3: デモ① 開発の効率化（約7分）

**スライド4 → Boomi画面に切り替え**

#### 実演の流れ

1. **Boomi Integration画面を開く**
   - 新規プロセスを作成する画面を見せる

2. **気象データ連携ロジックを構築（5分目標）**
   - HTTP Clientコネクタで気象APIを呼び出す設定
   - データ変換（JSON → Firebaseに送るフォーマット）をマッピング画面で作成
   - Firebaseへの書き込みコネクタを追加

3. **動作確認**
   - テスト実行し、データが流れることを確認

> **アピールポイント**: 「Firebase側のコードには一切手を入れていません。新しい連携が必要になるたびにコードを書くのではなく、ドラッグ＆ドロップで完了します。」

---

### パート4: デモ② API管理の松・竹・梅（約7分）

**スライド5 → Boomi APIM画面に切り替え**

お客様の予算感に合わせて、段階的な導入を提案する。

#### 梅プラン（スタート版）の実演

Firebase APIをBoomiでラップする手順を見せる。

1. **API Serviceコンポーネントを作成**
   - FirebaseのURLを「External Endpoint」として登録
   - 公開URLを `https://boomi-gateway.com/agri-api` に設定

2. **付加価値の実演**
   - **エンドポイント集約**: Firebaseの複雑なURLが `api.company.com/service` のようにシンプルになることを見せる
   - **APIキー認証**: Firebase側は何も変えずに、Boomi側でAPIキーを必須にする設定を行う
   - **ロギング**: APIが呼ばれた際に「誰がいつ呼んだか」がダッシュボードに即座に記録される様子を見せる

#### 松プラン（フル版）の紹介

画面を見せながら口頭で説明。

- **開発者ポータル**: APIの仕様書が自動生成され、外部開発者がセルフサービスでテストできる環境
- **高度な流量制限（Throttling）**: 特定ユーザーがAPIを叩きすぎてFirebaseの料金が跳ね上がることを防止
- **OAuth認証**: エンタープライズレベルのセキュリティを後付け

> **キーフレーズ**: 「まずは安く始めて、ビジネスの成長に合わせて管理機能をONにできます。」

---

### パート5: メリットとまとめ（約5分）

**スライド6〜7を使用**

製造・農業分野に特化した3つの価値を強調する。

| メリット | 具体的なシーン |
|---------|-------------|
| **コスト抑制** | 農繁期のアクセス集中でFirebaseの課金が爆発 → BoomiのThrottlingで防止 |
| **信頼性** | 農場からの不安定な通信、古い基幹システム → Boomiが吸収して安定稼働 |
| **資産継承** | 既存の手作りAPIを捨てずにモダンなアーキテクチャへ昇華 |

**クロージングメッセージ**:
「Boomiは単なる連携ツールではなく、**現場のスピードを殺さず、ITのガバナンスを守る**ための経営基盤です。」

## 想定Q&Aと切り返し

### Q1: 「APIMライセンスが高い」

**回答**: まずはIntegrationライセンスのみで始められます。HTTP Clientコネクタを使ってFirebaseをWebhookのように呼び出す構成なら、APIMなしでも基本的な統合は可能です。効果を実感してから、APIM機能を追加するステップアップが現実的です。

### Q2: 「Firebaseだけで十分では？」

**回答**: Firebase単体でも個別のAPIは作れます。ただし、APIが増えてきたときの横断的な管理（誰が・いつ・どれだけ使っているか）や、社内の基幹システム（SAP、RDBなど）との連携で大量のコードが必要になります。Boomiはその部分をノーコードで解決します。

### Q3: 「パフォーマンスへの影響は？」

**回答**: Boomi API GatewayはFirebaseの前段に入りますが、パススルーモードではミリ秒レベルのオーバーヘッドです。むしろ、Throttlingによって予期せぬアクセス集中からFirebaseを保護する効果があります。

### Q4: 「既存のFirebaseの設定を変える必要がある？」

**回答**: Firebase側は一切変更不要です。Boomiは外側からFirebaseのAPIを呼び出すだけなので、既存の認証やセキュリティルールはそのまま維持できます。

## 補足: さらに安く提案する場合

「APIMライセンスが高すぎる」という反応が出た場合の代替構成:

```
利用者 → Boomi Integration（HTTP Client） → Firebase API
```

- APIMなしで、IntegrationライセンスのHTTP Clientコネクタを使用
- Webhookのように動作させ、データ変換と簡易的なルーティングを実現
- まずはこの構成で効果を実感 → 後からAPIMライセンスを追加
